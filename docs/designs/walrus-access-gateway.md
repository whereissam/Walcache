## Architecture (from CEO review)

See full plan at: ~/.claude/plans/cosmic-prancing-quiche.md
See design doc at: ~/.gstack/projects/Walcache/huangbozhang-main-design-20260402-150835.md

Key architectural decisions:
- Gateway API is single Fastify server (acceptable for pre-adoption, plan for horizontal scaling)
- Access policies stored in SQLite via Drizzle ORM (not on-chain)
- NFT verification cached in Redis for 10 min (tradeoff: sold NFT retains access briefly)
- SealGate is optional policy type (fallback to authorization-only if Seal unavailable)
- @walcache/gate wraps @walcache/sdk (no code duplication)
- deployProtectedAssets uses saga pattern: create gate (pending) -> upload -> activate gate

## Critical Risks
1. Deploy-protected atomicity: upload succeeds but gate creation fails -> unprotected content
2. Seal key server dependency: if key servers down, encrypted content undeliverable
3. HMAC secret not rotatable without downtime
4. No developer interviews yet (The Assignment: talk to 5 Walrus devs before further investment)
